Unicode Library Design Document

Principles and objectives

  • Build on the existing C++ standard and standard library, including:

    • char, wchar_t, char16_t, char32_t, and their literals

    • string, wstring, u16string, u32string

    • std::locale

  • No modifications to existing standard library components.

  • Follow modern C++ design practices:

    • C++ 14 in general, the STL in particular

    • Generic programming

  • Do only enough to provide string interoperability and ICU integration. Then STOP!

Notes

  • Unicode algorithms may need more context than a single code point. For example, a (possibly lengthy) sequence of code points.

  • Unicode algorithms may need error handlers.

  • Unicode algorithms may need locales. Leave these algorithms to ICU!

  • Unicode algorithms may need property tables. Leave these algorithms to ICU!

Requirements

  • Follow an explicit and documented security model.

  • Error handling choices where appropriate.

  • Major components will supply both:

    • Low level generic components such as algorithms or iterators that focus on generality and wide use case coverage.

    • Higher level functionality that focus on convenience for the most common use cases.

  • Follow Unicode terminology (unicode.org/glossary/) for Unicode related terms.
